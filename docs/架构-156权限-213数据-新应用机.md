## 分工表

| 机器 | 放什么 | 不放什么 |
|------|--------|----------|
| **156** | OpenClaw；`permission_data.json` + `scripts/permission_vault_server.example.py`（只读 API） | 三羊 Flask、MySQL、Hermes 扛业务 |
| **213** | **MySQL 8** 全部业务表（见下表） | 3001/3002、OpenClaw、面板占大量内存 |
| **新应用机** | 双端 Python、Nginx、Redis（建议）、部署目录 `stable` | MySQL 进程、权威权限 JSON 写入 |

### 213 MySQL 里应有哪些（固定/主数据 + 订单缓存）

| 数据 | 表/模块 | 说明 |
|------|---------|------|
| 刀模库 | `dimoldb` | 已走 MySQL |
| 成品/原材料库存 | `inventory` 等 | 已走 MySQL |
| 原材料行 | `raw_materials` 相关 | 算料匹配 |
| 报价参数 | `quote_config` | 客服保存写入 |
| **订单缓存** | `order_cache_orders` / `order_cache_items` / `order_cache_meta` | **可以放，且应放 213**（`order_cache_store.py`） |
| 员工权限勾选 | `employee_permissions`、`employees`、`users` | 客服端 |
| 工单/流转 | `production_flows` 等 | 扫码报工 |

应用机通过 `.env` 的 `MYSQL_HOST=213内网IP` 读写以上全部数据，**不拷** `orders_cache.json` 到应用机（仅作迁移过渡）。

### 仍放在应用机上的文件（可接受）

| 文件 | 原因 |
|------|------|
| `.env`、`km_token.json`、`alibaba_shops.json` | 密钥，不进 Git，跟进程走 |
| `material_calc_cache.json` | 算料行级缓存，可后续迁表；体量小于订单库 |
| `static/`、`index*.html` | 由 Nginx 提供 |
| 日志 | 本地轮转 |

## 156：权限「通道断掉、应用只读」

应用机 `stable/.env`：

```env
PERMISSION_VAULT_URL=http://156内网IP:9443/permission_data.json
PERMISSION_VAULT_TOKEN=你的长随机串
# 不要配置 PERMISSION_VAULT_WRITE_URL
```

效果（代码已实现 `permission_vault.py`）：

- 启动时从 156 **GET** 拉 `permission_data`（工序、映射、模板等）
- 网页 **不能** 改工序/材质映射结构（生产端保存返回 403）
- 只有你在 156 上 SSH 改 JSON 或带 Token POST 到配置机

156 防火墙：

- 9443：**仅允许新应用机 IP**
- 22：仅你的 IP
- **不要**对公网开放 9443

OpenClaw 与 vault 可同机，但给 OpenClaw 单独限内存（systemd `MemoryMax`），避免拖垮 vault。

## 213：专机数据库

1. 安装 MySQL 8，绑定 **内网 IP**，`bind-address` 不对 0.0.0.0 公网开放。
2. 创建库 `sanyang`，用户：

```sql
CREATE USER 'sanyang_app'@'应用机内网IP' IDENTIFIED BY '强密码';
GRANT SELECT, INSERT, UPDATE, DELETE ON sanyang.* TO 'sanyang_app'@'应用机内网IP';
```

3. 从现网 213 导出再导入（若 MySQL 仍在旧 213 混跑机上）：

```bash
mysqldump -u... -p sanyang > sanyang_full.sql
# 在新 213 专机 mysql 导入
```

4. 213 上 **不要**再跑 `app_cs.py` / `app_production.py`；可保留 cron 做 `mysqldump` → OSS。

5. 安全组：3306 仅应用机安全组可入。

## 新应用机：只跑应用

1. 规格建议：**4核 8G**，SSD 80G+。
2. 同 VPC 与 213、156 内网互通。
3. `deploy.sh` 部署到 `/www/feijihe/stable`，systemd + venv（见 `deploy/systemd/`）。
4. `.env` 示例：

```env
MYSQL_HOST=213.内网.IP
MYSQL_PORT=3306
MYSQL_USER=sanyang_app
MYSQL_PASSWORD=...
MYSQL_DATABASE=sanyang

PERMISSION_VAULT_URL=http://156.内网.IP:9443/permission_data.json
PERMISSION_VAULT_TOKEN=...

FLASK_SECRET_KEY=...
```

5. **Redis（建议）**：装在本机，打单 dashboard 热数据；订单主数据仍在 213 MySQL。

## 订单缓存放 213 行不行？

**行，而且推荐。**

- 快麦同步后写入 `order_cache_store` → MySQL 表（不是只靠应用机本地 JSON）。
- 打单列表从 MySQL 读订单，应用机内存里再做 dashboard 聚合（可加 Redis 加速「秒开」体感）。
- 500～1000 单/天 + 子单行，MySQL 4C8G 专机足够；注意给 `order_cache_orders(so_id)`、`tid` 加索引（迁移脚本已建表）。

迁移注意：

- 若现网还有 `orders_cache.json`，用 `scripts/migrate_orders_cache_to_mysql.py` 灌进 213 后再关文件依赖。
- 应用机 **不要**再写本地大 JSON 覆盖 MySQL（`order_cache_store` 有空快照保护）。
