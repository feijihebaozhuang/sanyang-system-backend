# 权限配置放在独立服务器并锁定（仅你可改）

## 你现在仓库里有什么

| 内容 | 存在哪 | 谁能改 |
|------|--------|--------|
| 工序树、岗位、员工主数据、材质映射、权限模板 | `data.json` → `permission_data` | Admin 页面保存 → 写本机文件 |
| 每个员工每个功能的勾选 | 客服端 MySQL `employee_permissions` | 权限管理页保存 |
| Git / `deploy.sh` | **不会**覆盖服务器 `data.json` | 开发改代码碰不到线上 JSON |

你要的是：**`permission_data` 这一块放到另一台机器**，开发部署、Cursor 改代码都改不到，**只有你自己**能改。

代码已支持（`permission_vault.py` + `config_json.py`），按下面部署即可。

---

## 架构（推荐）

```text
┌─────────────────────┐         HTTPS + Token          ┌──────────────────────┐
│ 配置机（仅你可登录）  │  permission_data.json         │ 业务机 feijihe        │
│ 例如 10.0.0.5:9443  │ ─────────────────────────────► │ 3001 客服 / 3002 生产 │
│ SSH 只给你密钥       │         只读 GET               │ .env 只配 READ URL    │
└─────────────────────┘                                └──────────────────────┘
```

- **配置机**：放权威 `permission_data.json`，可用本仓库 `scripts/permission_vault_server.example.py` 或 Nginx 静态文件 + 防火墙。
- **业务机**：`stable/.env` 只配 `PERMISSION_VAULT_URL`，**不配** `PERMISSION_VAULT_WRITE_URL` → 页面保存工序/映射会提示已锁定，**deploy 不会写 permission_data**。

---

## 第一步：准备配置机上的 JSON

1. 从现网导出当前权威配置（业务机）：

```bash
# 在生产或客服 stable 目录
python3 -c "
import json
from pathlib import Path
d=json.loads(Path('data.json').read_text(encoding='utf-8'))
print(json.dumps(d.get('permission_data',{}), ensure_ascii=False, indent=2))
" > permission_data.json
```

2. 把 `permission_data.json` 拷到**配置机**，例如 `/opt/sanyang-config/permission_data.json`。

3. 配置机启动只读 API（示例脚本在仓库）：

```bash
cd /opt/sanyang-config
export PERMISSION_VAULT_TOKEN='你自定的长随机串'
python3 permission_vault_server.example.py
# 默认监听 0.0.0.0:9443
```

4. 防火墙：**仅允许业务机 IP** 访问 9443；**禁止**公网。配置机 SSH 只保留你的密钥。

---

## 第二步：业务机只读（3001 + 3002）

编辑 `/www/feijihe/stable/.env`（两台业务逻辑共用同一份 env 时写一次即可）：

```env
# 只读拉取权限配置（配置机内网地址）
PERMISSION_VAULT_URL=http://10.0.0.5:9443/permission_data.json
PERMISSION_VAULT_TOKEN=与配置机相同的随机串

# 业务机不要配置下面这行（留空 = 锁定，禁止网页改 permission_data 结构）
# PERMISSION_VAULT_WRITE_URL=
```

重启：

```bash
sudo systemctl restart sanyang-cs sanyang-production
```

验收：

- 权限管理里改「工序 / 材质映射」点保存 → 应提示 **已锁定在独立配置服务器**（生产端 `/api/permissions/save`）。
- `deploy.sh` 后员工权限勾选若仍走 MySQL，可继续变；**工序和映射以配置机 JSON 为准**（重启后生效）。

---

## 第三步：只有你能改配置

任选一种（可组合）：

1. **SSH 登录配置机**，直接改 `/opt/sanyang-config/permission_data.json`，重启配置机上的小服务或 touch 后由脚本重载。
2. **在本机浏览器**访问配置机内网管理页（若你把 `PERMISSION_VAULT_WRITE_URL` 只配在配置机本地的管理工具上，业务机不配 WRITE_URL）。
3. 配置机 `permission_vault_server.example.py` 的 **POST /permission_data** 需带 Token；仅你知道 Token。

改完后**无需改 Git**；业务机重启或等下次启动时会重新 `GET` 拉取（可在 `permission_vault.py` 后续加定时刷新，当前为**启动时 + read_permission_overlay 时**拉取）。

---

## 和「改权限」的关系

| 操作 | 配置机锁定后 |
|------|----------------|
| 开发 `git push` + `deploy.sh` | 只更新 `.py`，**不动** 配置机 JSON |
| Cursor 改仓库里 `data.json` | 仅影响本地；**线上以配置机为准** |
| 你在配置机改 JSON | ✅ 唯一权威来源 |
| 客服页勾选某员工「报价」 | 仍写 MySQL（若需一并锁定，可再关 `/api/permissions/save` 或只读库） |

---

## 环境变量说明

| 变量 | 业务机 | 配置机 |
|------|--------|--------|
| `PERMISSION_VAULT_URL` | 必填，GET 地址 | 可选 |
| `PERMISSION_VAULT_TOKEN` | 与配置机一致 | 必填 |
| `PERMISSION_VAULT_WRITE_URL` | **留空**（锁定） | 可设为 `http://127.0.0.1:9443/permission_data` 供 POST |

---

## 常见问题

**Q：配置机挂了怎么办？**  
业务机会回退读本机 `data.json` 里旧的 `permission_data`（日志里有 `保险库拉取失败，回退 data.json`）。建议配置机监控 + 备份 JSON。

**Q：现网还没 second server，先怎么锁？**  
继续用现网铁律：`deploy.sh` 不覆盖 `data.json`；`chmod 600 data.json` 属主 `admin`；仓库不提交真实 `data.json`。上配置机后按本文切换 `PERMISSION_VAULT_URL`。

**Q：报价、刀模会受影响吗？**  
不会。报价走 `quote_config` / `quote_data.json`；权限保险库只管 `permission_data`。
